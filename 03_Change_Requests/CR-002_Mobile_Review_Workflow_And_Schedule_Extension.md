# 项目变更单 CR-002

## 1. 基本信息

- 变更编号：CR-002
- 变更标题：新增企业上报、市省审核手机端能力，并将项目周期延长两周
- 提出日期：2026-04-10
- 提出来源：用户需求变更
- 提出人：项目干系人
- 审批人：PM
- 变更状态：已批准，需调整总计划

## 2. 变更内容

原计划仅聚焦企业填报与基础查询展示。现调整为：

1. 新增企业上报、市级审核、省级审核的完整业务链路。
2. 企业上报端和市、省审核端均需支持手机端使用。
3. 项目周期增加两周，由原定 2 个月调整为 2 个月加 2 周。

## 3. 变更原因

用户希望系统不仅完成企业数据采集，还能体现完整的上报审核闭环，并满足移动办公场景，以适应实际业务需要。

## 4. 影响分析

### 4.1 功能影响

1. 新增上报状态流转：草稿、已提交、市级审核中、市级退回、市级通过、省级审核中、省级退回、省级通过。
2. 新增市级审核和省级审核页面或工作台。
3. 新增审核意见、审核记录、状态变更记录。
4. 前端页面需要支持移动端布局与操作。

### 4.2 架构影响

1. 当前前端骨架需从“展示型首页”扩展为“多角色多页面系统”。
2. 后端需增加审核流接口、状态流转规则和权限区分。
3. SQLite 仍可继续作为课程版演示数据库，无需立即升级，但数据模型需要扩展。

### 4.3 计划影响

该变更属于重大变更，不是简单微调，原因如下：

1. 项目交付边界从“采集 + 查询”扩大为“采集 + 审核 + 移动端”。
2. 原 2 周计划不足以覆盖新增工作量。
3. 新增两周只是对现有正式项目周期的补充，项目应整体按“2 个月 + 2 周”进行阶段化规划。

### 4.4 风险影响

1. 审核流增加后，权限与状态控制复杂度明显上升。
2. 手机端适配若放到最后处理，返工成本高。
3. 若不重排里程碑，原有排期会失真，难以解释计划与实际差异。

## 5. 处理方案

1. 将项目总排期从“日计划”上升为“阶段 + 周计划”。
2. 当前已完成内容保留为“第 1 阶段基础骨架成果”。
3. 后续新增三大工作包：
   - 审核流程设计与实现
   - 移动端适配
   - 延长周期下的联调、测试与材料归档
4. 更新项目计划书、甘特图任务表、风险清单和里程碑。

## 6. 验收标准

1. 企业端可完成填报和提交。
2. 市级审核端可完成审核、退回和意见填写。
3. 省级审核端可完成终审和意见填写。
4. 手机端能完成企业上报和审核基本操作。
5. 项目计划与提交材料已反映周期调整和功能扩展。

## 7. 审批结论

审批结果：同意纳入项目范围，并据此修订总体计划。
