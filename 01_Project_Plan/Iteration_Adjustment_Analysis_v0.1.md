# 项目迭代调整分析 v0.1

## 1. 分析目的

本文件用于结合当前项目现状和两次正式需求变更，评估现有计划是否仍然适用，并给出修订后的项目推进安排。

## 2. 当前项目现状

截至当前，项目已完成以下阶段成果：

1. 项目计划 v0.1 和 v0.2 已完成。
2. 甘特图任务表已形成。
3. 前后端基础骨架已搭建完成。
4. 技术方案 v0.1 已形成。
5. README 初版已形成。

当前项目处于“基础骨架已就绪，核心业务尚未全面展开”的阶段，因此仍然具备较好的调整窗口。

## 3. 两次变更对项目的综合影响

### 3.1 变更一影响结论

“一至三月份半月报、其余月份整月报”会改变系统最核心的数据时间粒度，因此它不是纯展示层改动，而是会影响：

1. 数据模型。
2. 录入表单。
3. 校验规则。
4. 查询筛选。
5. 统计逻辑。
6. 导出格式。

结论：该变更需要尽早纳入核心业务开发，不适合放到最后补做。

### 3.2 变更二影响结论

“企业上报 + 市省审核 + 手机端 + 延长两周”会改变系统的业务边界和项目排期方式，因此属于重大变更。它新增的不只是界面数量，而是完整的角色链路和状态流转机制。

结论：这不是微调，而是需要将原来的 2 周开发节奏提升为阶段式迭代计划。

## 4. 是否需要微调项目迭代内容

结论：需要调整，而且第二次变更属于结构性调整，不应仅视为微调。

建议按以下原则修订：

1. 保留前 3 天成果，作为“第一阶段基础搭建”成果，不返工重做。
2. 将后续迭代重点改为“基础功能闭环 -> 报表周期规则 -> 审核流 -> 移动端适配 -> 联调测试 -> 材料归档”。
3. 项目总周期按“2 个月 + 2 周”重新解释和编排。

## 7. 对当前开发内容的直接调整建议

结合当前代码现状，建议立刻做以下调整：

1. 在数据模型中提前引入 report_type 和 report_period，避免后续返工。
2. 登录与权限实现时直接考虑企业用户、市级审核、省级审核三类角色。
3. 前端页面结构不要只做单首页，应尽早切换为多页面或多视图布局。
4. 审核流状态机应尽早定义，避免后续接口反复修改。
5. 手机端适配不要压到最后一周，应在页面开发阶段同步遵循响应式规范。

## 8. 结论

从当前整体情况看，项目前三步成果没有浪费，仍可作为正式基础资产继续沿用。但后续计划不能继续按原 14 天版本推进，必须切换为“阶段化 + 周度化”的新节奏。若继续沿用旧排期，会出现计划与实际需求严重不一致的问题。
