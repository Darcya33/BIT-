## 2. 项目背景

为满足课程作业中对项目计划、配置管理、变更控制和 Agent 编程软件系统的综合要求，本项目建设一个面向企业就业与失业数据采集的轻量级信息系统。系统模拟政府或行业主管部门对企业就业、失业信息的采集、上报、审核、查询和统计过程。

本项目不仅要求交付可运行系统，还要求提交项目计划、计划迭代历史、变更单、配置管理历史和演示材料，因此项目实施采用“文档与系统同步推进、每轮迭代均留痕”的方式开展。

## 3. 项目目标

### 3.1 总体目标

在课程作业周期内，完成一个可演示、可运行、可提交、可追溯的软件系统版本，并形成完整的软件工程过程材料。

### 3.2 具体目标

1. 实现企业就业与失业数据采集的核心业务闭环。
2. 支持企业上报、市级审核、省级审核的分角色业务流程。
3. 支持一季度半月报、其余月份月报的填报规则。
4. 支持桌面端与手机端的核心填报和审核操作。
5. 完成项目计划、变更控制、配置管理和演示材料的同步沉淀。

## 4. 项目范围

### 4.1 本期交付范围

1. 登录认证与多角色权限控制。
2. 企业基础信息管理。
3. 就业数据录入、查询与状态流转。
4. 失业数据录入、查询与状态流转。
5. 半月报与月报双报表模式。
6. 企业上报、市级审核、省级审核工作流。
7. 基础统计展示与状态概览。
8. 手机端基础适配。
9. 项目计划、变更单、CM 历史、README、演示脚本等交付文档。

### 4.2 本期不纳入范围

1. 外部系统集成。
2. 短信、邮件等复杂消息通知。
3. 生产级部署、高并发优化与安全加固。
4. 原生移动 App。
5. 大屏可视化和复杂 BI 分析。

## 5. 用户对象与角色定义

### 5.1 用户对象

1. 系统管理员：负责维护基础企业信息和查看整体运行情况。
2. 企业填报人员：负责录入就业失业数据、查看退回意见并重新提交。
3. 市级审核人员：负责企业上报数据的初审处理。
4. 省级审核人员：负责上报数据的终审处理。

### 5.2 角色权限边界

1. 管理员可查看总体概览并维护企业信息。
2. 企业用户仅能查看和维护本企业相关数据。
3. 市级审核员仅处理市级待审记录。
4. 省级审核员仅处理省级待审记录。

## 7. 需求变更与计划调整

### 7.1 CR-001 半月报规则变更

- 变更内容：1 至 3 月采用半月报，4 至 12 月采用月报。
- 影响范围：数据模型、录入表单、校验规则、查询统计口径。
- 处理结论：已纳入核心业务开发，并前置到填报能力设计中。

### 7.2 CR-002 审核流与移动端扩展

- 变更内容：新增企业上报、市级审核、省级审核，并要求手机端支持，同时项目周期延长两周。
- 影响范围：角色体系、工作流状态机、前端页面结构、里程碑与迭代节奏。
- 处理结论：项目由原先的短周期版本调整为阶段化推进版本。

## 8. 技术方案概要

### 8.1 技术选型

- 后端：FastAPI + Python 3.12
- 数据库：SQLite
- 前端：原生 HTML + CSS + JavaScript
- 配置管理：Git + GitHub

### 8.2 架构说明

1. 前端负责页面展示、视图切换与接口调用。
2. 后端负责角色校验、业务规则、数据存储和状态流转。
3. SQLite 用于课程演示版的数据持久化。
4. 系统采用轻量级前后端分离方式，便于快速迭代和演示。

## 9. 关键业务规则

1. 1 至 3 月仅允许半月报。
2. 4 至 12 月仅允许月报。
3. 企业记录先保存草稿，再执行上报。
4. 市级审核通过后进入省级审核。
5. 市级或省级退回后，企业可查看意见并重新提交。
6. 所有核心状态变更均需通过系统工作流接口执行。
